8.3   Component Description 
The principal compon ents of the system, with their responsibilities and salient

8.3.1   Web Client and Next.js Server 
The browser-side application is built with Next.js 14 using the App Router. The primary view 
is a chat surface modeled on the patterns established by widely -used assistant interfaces: a 
single scrollable transcript at the center, a composer at the bottom, and a side panel that 
surfaces structured artifacts (a cart preview, a product card, an order summary) when the 
agent's response includes them. The composer accepts free -form text, voice input via the 
browser's Speech Recognition  API (Chrome/Edge), and file uploads. Streaming responses 
from the U -AI are rendered token -by-token using the Server -Sent Events transport that 
Next.js supports natively. The application is intentionally minimalistic: there is no separate 
"products page" or "search page"; everything the user does happens through the chat. 
8.3.2   U-AI Runtime 
The U-AI runtime is a Node.js service exposing a single bi directional WebSocket endpoint 
(for chat) plus a handful of HTTP endpoints for housekeeping (authentication, profile 
retrieval, session listing). Internally, it composes the four logical modules described in 
Section 4.2 — intent extractor, context builder,  plan evaluator, response composer — into a 
per-session orchestration pipeline. Each user message triggers a fresh pipeline invocation; the 
pipeline maintains the running conversation context in an in -memory session table keyed by 
user id, with periodic persistence to MongoDB for recovery across restarts.

8.3.3   P-AI Runtime 
The P -AI runtime is a sibling Node.js service, deployed on the same private network but 
addressable only by the U -AI. It receives A2A messages, dispatches MCCA Requests to its 
plan synthesizer, returns Responses, and executes committed plans via MCP. The P -AI never 
receives raw user input; it sees only structured MCCA messages whose semantics have been 
distilled from user input by the U -AI. This is a deliberate security property — it limits the 
surface area for prompt-injection attacks that originate in user content. 
8.3.4   MCP Servers 
The six MCP servers enumerated in Section 7.7 each run as separate Node.js processes. They 
are managed in development by a process manager (PM2) and i n production by container 
orchestration; service discovery is performed by the P -AI at startup by reading a 
configuration file that lists the servers' endpoints. Each server is functionally a thin wrapper 
over the underlying data and services, registering its tools, validating inputs, executing the 
tool, and returning structured responses. 
8.3.5   Data Layer 
MongoDB Atlas serves as the primary data store. The schema is documented in detail in 
Chapter 9. The choice of MongoDB rather than a relational store r eflects the schema's 
document-oriented nature: products have varying attribute sets across categories (size matters 
for shoes but not for books; battery life matters for phones but not for shirts), orders have 
nested line items with their own evolving stru cture, and user profiles aggregate 
heterogeneous data (preferences, prior orders, address book, payment methods). The 
flexibility of a document store is a better match for this shape than a normalised relational 
schema would be.

8.6   Scalability Considerations 


=== PAGE 77 ===
Human–Computer Interaction through Context-Aware Collaborative AI Agents 
59 
 
While the present project is a laboratory prototype, the architecture has been chosen with 
scaling in mind. Several properties of the design either enable scaling out of the box or ma ke 
it tractable with modest additional work. 
Statelessness at the agent layer. The U -AI and P -AI runtimes are stateless across 
requests: all per-session state lives either in MongoDB (persistent) or in an in -memory cache 
that can be reconstructed on demand . This makes horizontal scaling straightforward: 
additional agent runtime instances can be added behind a load balancer with no special 
coordination. 
Independent scaling of components. Because each component is a separate process 
communicating through type d contracts, components can be scaled independently. The 
catalog-mcp, which handles the highest query volume, can be replicated to ten or twenty 
instances; the payments -mcp, which handles lower volume but higher criticality, can be 
replicated to three high ly-available instances. The U -AI and P -AI runtimes can be scaled in 
proportion to their respective LLM costs, which may differ. 
LLM-call caching. A significant fraction of intent extraction calls are repeat or near -
repeat queries (the user types the same k ind of request multiple times within a session, or 
different users issue similar requests). A semantic cache at the U -AI layer, keyed on 
embedding similarity to recent prompts, can absorb a substantial fraction of these calls 
without LLM round -trips, reduc ing both latency and cost. The current implementation 
includes a primitive exact-match cache; a semantic cache is on the roadmap. 
Tool-call batching. Several MCCA plans involve calling the same tool with several 
different argument sets (e.g., fetching deta ils for three candidate products in parallel). The 
MCP client supports parallel dispatch and the underlying MongoDB queries are independent, 
so this batching is effectively free. The current implementation issues parallel tool calls when 
the plan synthesizer flags them as parallelizable. 
Failure isolation. A failure or slowdown in a single MCP server affects only the tools 
that server hosts. The P-AI can, in principle, plan around the failure (e.g., if payments -mcp is 
unavailable, the P-AI can complete a "s ave for later" flow but not a "buy now" flow, and the 
U-AI can narrate this gracefully). The current implementation surfaces such failures to the 
user via the U-AI; future work includes more sophisticated graceful-degradation logic.

import { NextRequest } from 'next/server'; 
import { getServerSession } from 'next-auth'; 
import { authOptions } from '@/lib/auth'; 
import { runUserAgent } from '@/lib/agents/u-ai'; 
export const runtime = 'nodejs'; 
export async function POST(req: NextRequest) { 
  const session = await getServerSession(authOptions); 
  if (!session?.user?.id) { 
    return new Response('Unauthorised', { status: 401 }); 
  } 
  const { message } = await req.json(); 
  const stream = new ReadableStream({ 
    async start(controller) { 
      const encoder = new TextEncoder(); 
      try { 
        for await (const token of runUserAgent(session.user.id, message)) { 
          controller.enqueue(encoder.encode(`data: 
${JSON.stringify(token)}\n\n`)); 
        } 
        controller.enqueue(encoder.encode('event: done\ndata: {}\n\n')); 
      } catch (e: any) { 
        controller.enqueue(encoder.encode(`event: error \ndata: 
${JSON.stringify({ message: e.message })}\n\n`)); 
      } finally { 
        controller.close(); 
      } 
    }, 
  }); 
  return new Response(stream, { 
    headers: { 
      'Content-Type': 'text/event-stream', 
      'Cache-Control': 'no-cache', 
      'Connection': 'keep-alive', 
    }, 
  }); 
}
